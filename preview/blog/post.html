Modernizing Legacy ASP Applications to Blazor

A deep dive into the challenges and lessons learned from migrating classic ASP applications to modern Blazor WebAssembly on Azure

The Challenge

When we started the migration project, we were dealing with Classic ASP applications that had been running production for over 15 years. These weren't simple brochure sites—they were critical engineering tools used daily across multiple manufacturing facilities.

The biggest challenge wasn't the technology itself, but maintaining business continuity. We couldn't afford downtime, and we couldn't risk losing functionality that engineers depended on for their daily work.

Planning the Migration

We broke the migration into three phases:

  1. Assessment and Architecture - Understanding what we had and designing the target state
  2. Incremental Migration - Moving functionality piece by piece, testing thoroughly
  3. Optimization - Taking advantage of modern platform capabilities

This phased approach let us learn from early mistakes and adjust our strategy before tackling the most complex parts of the system.

Technology Decisions

We chose Blazor WebAssembly for several reasons:

  • Our team already knew C# well from backend development
  • We could share models and logic between client and server
  • The component model was a good fit for our UI patterns
  • It integrated naturally with our existing .NET infrastructure

Key Lessons Learned

1. Don't Translate, Rethink

Our first instinct was to port the old code directly. This was a mistake. The old applications had grown organically over 15 years, accumulating workarounds and technical debt. A direct port would just move that debt to a new platform.

Instead, we used migration as an opportunity to rethink workflows and simplify where possible. Some features that seemed critical turned out to be rarely used and could be removed entirely.

2. State Management is Critical

Classic ASP applications relied heavily on server-side session state. Moving to a client-side SPA meant rethinking how we handled state. We ended up with a hybrid approach—some state in the browser for responsiveness, some server-side for security and consistency.

3. Performance Testing Under Load

Blazor WebAssembly has a different performance profile than server-side applications. The initial load is heavier (downloading the .NET runtime and app DLLs), but subsequent interactions are faster. We had to adjust user expectations and optimize the initial load experience.

The Results

After 18 months of work, we've successfully migrated three major applications. The benefits have been significant:

  • Faster development cycles for new features
  • Better reliability and easier debugging with modern tooling
  • Improved user experience with responsive, interactive UIs
  • Easier onboarding for new developers

The investment was substantial, but the modernized applications are easier to maintain, more reliable, and positioned for future enhancements.

Looking Forward

The migration taught us that legacy modernization isn't just about new technology—it's about understanding why systems evolved the way they did and making deliberate choices about what to preserve and what to rebuild.

If you're considering a similar migration, my advice is to start small, learn from each phase, and don't underestimate the importance of having buy-in from the people who will actually use the applications.

Example: Shared Model Pattern

One pattern we found particularly useful was sharing models between client and server: 

// Shared/Models/WorkOrder.cs
public class WorkOrder
{
    public int Id { get; set; }
    public string OrderNumber { get; set; }
    public DateTime CreatedDate { get; set; }
    public OrderStatus Status { get; set; }

    // Validation logic shared by client and server
    public bool IsValid() =>
        !string.IsNullOrEmpty(OrderNumber) &&
        CreatedDate > DateTime.MinValue;
}

This eliminated duplication and ensured consistent behavior across the application.
